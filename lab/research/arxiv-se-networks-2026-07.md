## Research findings: arXiv-mined structural leaps for deterministic code retrieval

I ran targeted searches across arXiv/ACM/Springer for the five hunt areas and cross-checked abstracts (not just search snippets) for the highest-value hits. Below: full candidate list by hunt area, then a ranked top 5 against G1–G4.

### 1. Community detection on dependency graphs (→ G1)

- **Traag, Waltman & van Eck, "From Louvain to Leiden: guaranteeing well-connected communities"** (arXiv:1810.08473, Sci. Reports 2019). Leap: Louvain can output internally-*disconnected* "communities" (a silent correctness bug for module recovery); Leiden's refinement phase guarantees every returned community is connected and well-separated, at comparable cost. Mechanism: run Leiden on the import graph; each community becomes a "search first here" module before lexical ranking runs within it.
- **GVE-Leiden** (arXiv:2312.13936, 2023). Leap: shared-memory parallel Leiden that clusters graphs with millions of edges in near-real-time. Mechanism: makes "cluster fresh on every large repo, don't precompute/cache" affordable even at monorepo scale (10k+ files → tens of thousands of import edges is trivial for this).
- **"Using Structural and Semantic Information to Identify Software Components"** (arXiv:2102.04710) — compares Leiden/Louvain/Infomap directly for software component identification; confirms Leiden-family algorithms are the applied standard in this niche.
- **Microservice/service-identification literature** (ServiceMiner and comparable bottom-up structural clustering): **77.9% precision / 66.4% recall / 71.7% F-measure** identifying architecturally-significant service boundaries purely from static structure. This is the closest quantitative proof-point that "find the right module first" is a tractable, measurable sub-problem with today's classical clustering — directly derisks a module-then-refine architecture for G1.

### 2. Static interface/dispatch bridging, no execution (→ G2)

- **"Scalable and Precise Application-Centered Call Graph Construction for Python" (Jarvis)** (arXiv:2305.05949). Leap: flow-sensitive type inference *per function* via local Function Type Graphs (copy-on-branch, merge-on-join) instead of one global flow-insensitive assignment graph — gets flow sensitivity without the usual whole-program blowup. Measured: **84% higher precision, ≥20% higher recall, 67% faster than PyCG**; PyCG failed outright (OOM/timeout/recursion) on all 6 real-world macro-benchmark apps (100k–515k whole-program LOC) where Jarvis finished in ~8s/227MB average. This is a direct, load-bearing citation for G2's measured Python misses: it's a concrete, cheap, deterministic recipe for resolving "which implementation does this call/interface actually reach" statically.
- **SkipFlow** (arXiv:2501.19150). Leap: lightweight interprocedural points-to analysis that also tracks *primitive values* (not just object refs) and explicit branch/predicate edges, cutting spurious reachable-method edges by **9% on average** with negligible time overhead. Mechanism: a cheap precision knob on top of Andersen/Steensgaard-style analysis for pruning interface-dispatch over-approximation.
- **Class Hierarchy Analysis (CHA)** (Dean, Grove, Chambers, ECOOP'95 — foundational but still the workhorse floor). Leap: compute the full set of concrete implementations reachable from an interface/virtual call using *only* the class/interface hierarchy, no dataflow at all. Mechanism: cheapest possible fallback for Go interfaces — index every interface's satisfying types, and when an issue's vocabulary matches an interface method, fan out to all satisfying implementations as candidates before/alongside lexical scoring.

### 3. Spectral/centrality beyond PageRank (→ G3)

- **Chung, "The heat kernel as the pagerank of a graph"** (PNAS 2007, foundational; still actively used, e.g. arXiv results on heat-kernel PageRank for local clustering). Leap: heat-kernel diffusion is a one-parameter (diffusion time *t*) generalization of personalized PageRank that is provably better at finding *tightly-linked local clusters* rather than globally popular nodes — the opposite failure mode from plain PageRank (which tends to crown generic "hub" files, e.g. `utils.py`, as central). Mechanism: score files with heat-kernel PageRank at small *t* seeded from lexical hits; low *t* biases toward the *nearest* structurally-tight neighborhood rather than the globally-popular file.
- **"Efficient Algorithms for Computing Random Walk Centrality"** (arXiv:2510.20604). Covers Katz centrality, commute-time (resistance) distance, and random-walk betweenness with scalable algorithms. Leap: resistance distance measures "how hard is it to avoid this node" (redundant-path robustness) rather than "how many walks pass through it" — a structurally distinct notion of centrality from PageRank's stationary-distribution popularity. Mechanism: use resistance-distance-to-seed as a second ranking feature; a file that's on the *only* low-resistance path from an issue's symptom to a subsystem is a strong "the" fix-location signal even with modest PageRank.
- **struc2vec** (arXiv:1704.03165, KDD'17) — learned, so out of scope for a no-ML tool as-is, but its underlying primitive (ordered degree-sequence fingerprints at increasing k-hop rings = a role signature) is fully computable deterministically without the word2vec step. Mechanism: compute a deterministic "role vector" (in/out-degree histogram at hop 1, 2, 3) per file; files whose role vector matches the canonical "interface implementation" or "orchestrator" shape get a boost — useful jointly with G2.

### 4. Non-neural bug/feature localization 2023–2026 holdouts (→ G4)

- **Maarleveld, Guo & Feitosa, "Gotta catch 'em all! Towards File Localisation from Issues at Large"** (arXiv:2507.18319, 2025). This is the most important G4 citation. It deliberately builds a *classical IR baseline* (TF-IDF/BM25-family) across **all** issue types, not just bugs, and finds: (a) bug-specific heuristics from the localization literature generalize poorly to non-bug issues; (b) **identifier presence has only a small effect on performance for most issue types** — i.e., the "NL-only issue" ceiling isn't purely an identifier-absence problem, it's a genuine vocabulary/semantic gap that pure lexical scoring can't close; (c) results are strongly project-dependent, arguing for per-project tunable retrieval rather than one global recipe. Leap: this is direct evidence (from 2025, non-neural) that G4 is a real ceiling for lexical methods and *confirms your measured number is the field's number*, not an implementation gap in bgrep.
- **"An empirical assessment of baseline feature location techniques"** (Empirical SE, 2020; 8 techniques × 12 case studies). Leap: rigorously shows VSM-Lucene and LSI outperform other classical FLTs, and that technique ranking is corpus-dependent — a calibration reference for which classical baseline to benchmark bgrep against.
- **Poshyvanyk & Marcus, conceptual coupling metrics** (foundational, still cited as the non-neural structure+text hybrid standard) — combines structural (call/data coupling) and textual (LSI) similarity into one deterministic coupling score; a template for fusing bgrep's lexical and graph scores into a single metric rather than a two-stage filter.

### 5. Wildcards

- **Koopman, "The Theory of Search"** (Operations Research 1956–57, foundational) + modern treatments (arXiv:2206.06450 "On several properties of uniformly optimal search plans"). Leap: given a prior probability-of-containment map and an exponential detection function, the optimal allocation of search effort **equalizes marginal detection-probability-per-unit-effort across cells**, not uniform coverage and not pure rank-order greedy. This maps almost exactly onto "given a token budget and a relevance prior over files/modules, how many tokens to spend reading each candidate" — a formal justification for a water-filling token allocator instead of bgrep's presumed top-k-then-truncate.
- **Charnov's Marginal Value Theorem** (1976, foundational ecology, still the base model in active 2024–2025 foraging papers e.g. bioRxiv:2025.07.04.663150 on MVT under energetic constraints). Leap: a forager should leave a patch exactly when its instantaneous yield rate drops below the *environment's average* yield rate — a clean, parameter-light stopping rule. Mechanism: treat each candidate file/module as a "patch"; while expanding it (reading more lines, following more imports) yields new-relevant-tokens-per-token-spent above the running corpus-wide average, keep expanding; the moment it dips below, stop and move to the next candidate. This is a one-line deterministic budget-allocation policy with a 50-year empirical track record, directly reusable as-is.
- **Lawrance, Burnett, Bogart, Bellamy & Rector, "How Programmers Debug, Revisited: An Information Foraging Theory Perspective"** (TSE 2013) and **Niu et al., "Foraging and navigations, fundamentally: developers' predictions of value and cost"** (FSE 2016). Leap: navigation during debugging is well-predicted by "scent" (local textual/structural similarity to the goal) and "topology" (graph distance) alone — no need to model hypotheses/mental state — measured on real debugging sessions with professional programmers. Mechanism: this is the theoretical grounding for exactly the kind of scent-following graph-walk bgrep already does; it also supplies a *cost* term (patch residence time) that pairs naturally with MVT above.
- **npm/Cargo dependency-vulnerability propagation studies** (arXiv:2201.03981, arXiv:2604.17668) — not percolation theory per se, but empirically measured "blast radius" distributions (how many hops of transitive dependents are typically affected) on graphs of 1M+ packages. Leap: gives calibrated priors for how far change-impact should be expected to propagate through real dependency graphs — useful as a sanity-check/prior for G2's impact-set sizing rather than a new algorithm.
- Direct **percolation-theory-on-code** work (edge/bond percolation to find natural module cut-points) did not surface a software-specific paper with measured results — flagging this as a plausible but currently *unvalidated* transfer, not a confirmed leap.

One directly on-target existing-tool citation surfaced during the hunt (not requested but load-bearing for calibration): **"How Much Static Structure Do Code Agents Need? A Study of Deterministic Anchoring"** (arXiv:2606.26979, ISSTA 2026). Even though it's evaluated through an LLM agent (Codex), the *signal itself* is fully deterministic — injecting call/inheritance/config topology as structural facts. Measured: **+2.2pp Func@5, -1.6 interaction rounds, link-following rate 0.15–0.18 → 0.21–0.24, run-to-run variance roughly halved, Pass@1 +3.4pp**, at ~10% more tokens; critically, it also found **hub-heavy repos benefit from inverse-only edges** (who-calls-me, not forward edges) — a direct, measured design rule for G1/G3 at monorepo scale.

### Ranked top 5 by expected impact-per-cost against G1–G4

1. **Leiden (+ GVE-Leiden for scale) module-first funneling — G1.** Highest leverage because G1 is the architecturally blocking gap (everything downstream operates inside whatever module you pick first), the algorithm is mature/off-the-shelf, connectivity-guaranteed (fixes Louvain's silent bug), and sub-second at monorepo scale. Cost: low (one library call plus a two-phase retrieval loop).

2. **Search-theoretic token-budget allocation — Koopman optimal-effort-density + Charnov's Marginal Value Theorem — cross-cutting G1–G4.** Near-zero implementation cost (it's an allocation *policy*, not new infrastructure), and it improves the payoff of every other signal you already compute by spending tokens where marginal relevance-per-token is highest instead of flat top-k truncation. Best cost-adjusted ROI of anything found.

3. **Jarvis-style flow-sensitive-lite dispatch resolution (+ CHA floor for Go) — G2.** Directly targets the measured interface/indirection miss with a published, reproducible precision/recall win (84%/20%+ over PyCG) and known scaling behavior (~8s/227MB on 100k+ LOC apps). Cost: moderate — needs a per-language static pass, but Python and Go both have tractable recipes (Jarvis; CHA-over-interfaces respectively).

4. **Heat-kernel PageRank / resistance-distance as a second centrality signal fused with PageRank — G3.** Attacks the exact weak metric (File@1=.354) by correcting PageRank's bias toward generic hubs; heat-kernel at small diffusion time favors tightly-linked local clusters, resistance distance favors "only path" files. Cost: moderate (one extra sparse linear-algebra pass), well-understood math, no training.

5. **Deterministic structural-anchor features (call/inheritance/inverse-only-for-hubs) as first-class ranking signal — G1/G3.** The ISSTA 2026 anchoring paper gives measured evidence this specific signal shape (and specifically inverse-edges for hub-heavy repos) improves localization and stability; bgrep likely already extracts most of this via tree-sitter, so it's mostly a re-weighting/feature-engineering cost, not new infrastructure.

**G4 note:** no method found — neural or non-neural — cracks NL-only issue localization head-on; the strongest 2025 non-neural baseline paper (arXiv:2507.18319) confirms the ~35-40% ceiling is a field-wide property of lexical methods, not an implementation gap. The evidence points to attacking G4 *indirectly*: shrink the search space with #1 so a weak lexical signal only has to discriminate within a small module, and spend disproportionate budget on ambiguous candidates via #2 rather than looking for a silver-bullet G4-specific signal.
