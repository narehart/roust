# Fault Localization / Root-Cause / Source-Detection Literature Scan

## Ranked top 3 for bgrep's residual failure mode (vague-NL issue, correct file loses to same-directory siblings)

### #1 — Call-graph PageRank boosting of spectrum scores → target: **within-subsystem, @1**

"Boosting Spectrum-Based Fault Localization via PageRank" (Zhang et al., ISSTA 2017, static call/dependency graph over program entities weighted by SBFL suspiciousness, then propagated with PageRank) and its 2019 follow-up empirical study (Lingming Zhang's group) show that re-ranking raw SBFL suspiciousness scores by propagating them across a **static call graph** materially improves ranking, especially for breaking ties among structurally similar candidates. The mechanism: build a directed graph of call/import edges among candidate files; seed each node's suspiciousness from the base score (in bgrep's case: text-similarity/BM25 score); run PageRank (or a bounded-hop propagation) so files that are *actually depended on or actually depend on* the entities named in the query gain rank relative to siblings that just happen to sit in the same directory but are structurally disconnected from the mentioned symbols.

**Deterministic mechanism sketch for bgrep**: build a static import/call graph at index time (no test execution needed — this is exactly the "static call graph as spectrum" idea from the coverage-approximation literature below). At query time, seed PageRank weights from bgrep's existing lexical/structural match scores, propagate 1–2 hops, re-rank. This is a pure graph computation, no LLM, no training — directly attacks the sibling-tie problem because siblings that aren't in the call/import path get down-weighted relative to the one file that's actually wired into the mentioned functionality.

### #2 — Rumor-source / rumor-centrality estimation on the dependency graph → target: **within-subsystem, NL-only**

Shah & Zaman, "Rumors in a Network: Who's the Culprit?" (arXiv:0909.4370) and "Finding Rumor Sources on Random Trees" (arXiv:1110.6230, *Operations Research* 2016) define **rumor centrality**: a graph-score function counting the number of distinct spreading orders consistent with a node being the source, proven to be the maximum-likelihood estimator for source detection under the SI (susceptible-infected) model on regular trees. Key numbers: detection probability caps at ~31% under MLE even on regular trees (α₃=0.25 for 3-regular trees, 0<α_d<0.5 for d≥4), and drops below 5% on realistic scale-free/Facebook/AS-graph topologies — so as a *hard localization* tool it's weak, but its real value is as a **tie-breaking / relative-ranking signal among symmetric candidates**, which is precisely bgrep's failure mode. Rumor centrality is a purely combinatorial computation (counting linear extensions of a partial order via a message-passing recursion over the tree/graph) — no learning, closed-form, O(n log n) on trees.

**Deterministic mechanism sketch**: treat the set of files/symbols explicitly named or textually co-occurring in the issue as the "infected set" on the static import graph. Compute rumor centrality (or a bounded local approximation) restricted to the subgraph spanning that infected set plus their same-directory neighbors — the candidate whose position in the dependency graph maximizes the number of consistent "spreading orders" from the issue's mentioned entities is the most probable true source file, even when several siblings look identical by pure text/path similarity. This is the most literal transfer from epidemic source detection to "which file is the source of this misbehavior given the modules the issue mentions" — but it's unvalidated in the SE domain (no paper applies rumor centrality to code); treat as a research bet, not an established technique.

### #3 — SBEST: stack-trace/mention-reachability without failing tests → target: **coverage, @1 (MAP/MRR)**

Kabadi et al., "SBEST: Spectrum-Based Fault Localization Without Fault-Triggering Tests" (arXiv:2405.00565, accepted *Empirical Software Engineering* 2025). Core empirical finding directly transferable to bgrep: **78.3% of buggy methods are reachable within an average of 0.34 method calls from the stack-trace frame**, and 98.3% of bug fixes directly address the exception/entity named in the trace. SBEST combines this reachability signal with ordinary coverage/spectrum data (when available) to rank suspicious locations without needing an actual failing test; reported gains are +32.22% MAP and +17.43% MRR over a stack-trace-only baseline (mean MAP 0.428, MRR 0.496). It's deterministic SBFL-formula application (Tarantula/Ochiai/D*) over the coverage+trace union — no ML.

**Deterministic mechanism sketch for bgrep (test-free, NL-only analog)**: bgrep issues don't have stack traces, but they often name a symbol, error message, or config key. Treat any explicitly named identifier/path in the issue as a "trace frame" and compute static reachability distance (in call-graph hops) from that anchor to each candidate file, exactly mirroring SBEST's 0.34-hop empirical finding — i.e., weight candidates by `1/(1+hops)` from the nearest named anchor rather than by raw text similarity alone. This is the cleanest of the three to implement because it needs no graph-theoretic machinery beyond BFS on the existing import graph, and it directly targets the "vague NL issue, real answer is one hop from something the issue *does* name" pattern.

## Other findings, by topic (not top-3 but relevant)

**1. SBFL state of the art / file-level accuracy.** A Dec-2024 empirical evaluation reports SBFL Top-1 = 47.6%, Top-5 = 59.6%, Top-10 = 64.6% on real-world faults ([IJRISS 2024](https://ideas.repec.org/a/bcp/journl/v8y2024i12p439-444.html)) — Ochiai remains the best individual formula (Abreu et al., confirmed by Santelices et al.). These numbers are for **statement-level**, dynamic-coverage-requiring SBFL — a ceiling reference, not directly applicable since bgrep has no test execution.

**2. Static approximation of coverage.** "Static Estimation of Test Coverage" (Ball et al.) — uses static call-graph reachability as a stand-in for dynamic coverage; precision is bounded by call-graph extraction precision, and imprecision comes mainly from calls under untaken conditional branches (I could not retrieve exact precision numbers — ResearchGate returned 403). This validates the general approach behind #1 and #3 above: a **static call/import closure as a "spectrum"** is an established, if imprecise, substitute for dynamic coverage.

**3. Coverage-based localization without failing tests / test-as-query.** IR-based fault localization (BLUiR and successors) already treats the bug report as a query and source files as documents via structured text similarity, sometimes combined with a call-graph score in learning-to-rank models — this is architecturally close to what bgrep does already; the delta is adding the call-graph score as an explicit deterministic feature rather than relying on text alone.

**4. MBFL cost/benefit.** Mutation-based FL (MUSE, Metallaxis, BLMu) needs full test-suite re-execution against each mutant — "very costly and time-consuming," and in the worst case has *worse* EXAM scores than SBFL despite the cost. Not viable for bgrep's execution-free, sub-second-latency design goal; skip.

**5. SRE/microservice RCA.** TraceRCA (Li et al., 2021) — deterministic, non-ML — mines the microservice set with highest Jaccard-index concentration of abnormal traces, reported ~83% top-1 accuracy on average, beating unsupervised baselines by 51–81% on larger systems ([netman.aiops.org PDF](https://netman.aiops.org/wp-content/uploads/2021/05/1570705191.pdf)); MicroRCA/graph+PageRank variants reach ~92% precision. Structurally this is the same "propagate anomaly signal over a static dependency graph" idea as #1, just from ops rather than SE — reinforcing that graph-propagation over dependency structure is the dominant deterministic technique in adjacent fields.

**6. Delta debugging / static causal reduction.** "Reduce Before You Localize" shows delta-debugging a failing input *before* running SBFL improves localization accuracy, but requires re-execution to check reduced inputs still fail — not applicable statically. Program slicing (static backward/forward slices) is the one delta-debugging-adjacent technique usable without execution, and has precedent combined with SBFL (intersection of static slice + dynamic execution slice); could be a fallback filter (restrict candidate file set to the static slice from issue-named symbols) but adds complexity for uncertain gain.

**7. 2023–2026 non-LLM FL.** The strongest non-LLM recent work is GNN-based: DepGraph (FSE 2024) — reduces graph complexity 70% and beats prior GNN-FL (Grace) by 20% Top-1, 42% Top-1 in cross-project settings; GraMuS (2025) similarly graph+multimodal. These require trained models (not deterministic/training-free) so they're a ceiling reference rather than something bgrep can adopt directly, but they empirically confirm that **graph-structural features (not just text) are what closes the gap on hard cases** — consistent with recommending #1–#3 above.
