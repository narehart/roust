# Function/Line-Level Localization: How SOTA Systems Narrow, and What's Transplantable

*Companion to `lab/research/wave1/fault-localization.md`. Question: given roust beats Agentless at FILE (92.3 vs 69.7) but loses at FUNCTION (41.0 vs 52.0), what mechanical step converts a correct file into a correct function/line for published systems, and how much of that step is deterministic scaffolding vs LLM judgment?*

## Ranked top 3 for roust's function/line gap

### #1 — Function/class-boundary-aware candidate units (Agentless's AST skeleton, repurposed as an indexing unit rather than a summarization format) → target: **function-level exact match**

Agentless (arXiv:2407.01489) builds its "skeleton" by parsing every file's AST and unparsing it into a canonical headers-only form (class/function signatures + docstrings, bodies stripped) — this construction step is **100% deterministic**, confirmed by direct quote: *"we begin by parsing both the old and new code…into abstract syntax trees. Next, we unparse the trees into a canonical source code format."* Only the *selection* over that skeleton ("which classes/functions look suspicious") is LLM judgment; their own ablation shows the deterministic skeleton format alone (vs. raw file content) lifts Contains-GT from 53.67% to 58.33% at equal or lower cost — i.e., part of the gain is a **representation effect**, not a reasoning effect. **Transplant verdict: high value, trivial effort.** roust currently packs line-*regions* (arbitrary windows chosen by greedy coverage), not function-*boundaries*; if the eval unit is "does the packed region equal the gold function," a pipeline that scores and packs at AST-function granularity (one candidate = one function body, boundary-snapped) should mechanically raise FUNCTION exact-match even with zero change to the underlying ranking signal, because right now a correct-but-mis-bounded region can straddle a function edge and lose credit it structurally deserves.

### #2 — Entity-granular deterministic search index (LocAgent's SearchEntity — validated as the dominant lever, *not* the graph traversal) → target: **function-level Acc@5/@10**

LocAgent (arXiv:2503.09089) builds a heterogeneous graph via deterministic tree-sitter/AST parsing (files→classes→functions, contains/invoke/import/inherit edges), then exposes three tools to the LLM agent: `SearchEntity` (exact-ID → fuzzy-name → BM25 inverted index, all deterministic retrieval), `TraverseGraph` (deterministic BFS, LLM only picks hop-count/direction/relation-type), and `RetrieveEntity` (deterministic fetch). The critical finding is in their own ablation (Qwen-2.5-7B-ft, file Acc@5 baseline 88.32%): removing `SearchEntity` costs **-19.34%**, collapsing to BM25-only costs **-13.14%**, but removing `TraverseGraph` (the actual graph-hopping part) costs only **-2.19%**, and fixing traversal hops at 1 costs **-1.46%**. In other words, the graph/agentic machinery that looks like the "LLM reasoning" contribution is mostly cosmetic — **the entity-granular BM25 index over functions/classes (not files) is where almost all the deterministic lift lives**, and it's already a pure inverted-index computation. **Transplant verdict: highest expected gain per unit effort.** This is the single most actionable transplant: build roust's existing lexical/symbol-weighted index at function-chunk granularity (seeded from the same AST parse as #1) instead of whole-file or arbitrary-window granularity, and query it directly — no graph traversal needed to capture most of the effect LocAgent attributes to its "agent."

### #3 — Deterministic graph-distance pruning/weighting (OrcaLoca's CodeGraph shortest-path metric; CoSIL's call-graph BFS scaffold minus its LLM pruner) → target: **@1 tie-breaking within a file**

OrcaLoca (arXiv:2502.00350) computes a genuinely deterministic distance-aware pruning score, `d(SR,PB) = (1/|PB|) Σ min(d(v_SR,v), d(v,v_SR))`, a shortest-path average over a CodeGraph of containment + call-reference edges, used to keep the top-k=12 nearest entries — this piece needs no LLM at all. CoSIL (arXiv:2503.22424) similarly does deterministic module/function call-graph construction and BFS expansion, but its "pruner" (the node-relevance filter that decides which BFS branches survive) is an **LLM boolean call**, and their ablation shows the *iterative search* (i.e., the graph scaffold + expansion loop, largely deterministic) contributes the most (-27.8% if removed) while the LLM pruner itself contributes comparatively little (-4.6%). **Transplant verdict: moderate, and already on roust's roadmap** (`ROADMAP.md` items #1/#3/#4 — call-graph PageRank, CHA, heat-kernel diffusion — independently converge with three separate papers here on "static call/reference graph distance from named anchors" as a deterministic re-ranking feature). Expected gain is secondary to #1/#2 because it's a tie-breaker on top of an already-correct candidate set, not a fix to the candidate-set/granularity mismatch itself.

## Per-system dissection

### Agentless — arXiv:2407.01489
**Narrowing:** three explicit stages — (1) file-level: LLM ranks repo tree + optional embedding retrieval, combined by intersection; (2) element-level: LLM selects suspicious classes/functions from the deterministic AST-skeleton of files chosen in (1); (3) edit-location: LLM reads full code of selected elements and emits line/function/class edit locations, sampled 4x. **Deterministic vs LLM:** skeleton *construction* (AST parse → canonical unparse) is deterministic; every *selection* decision (which files, which elements, which lines) is LLM judgment. Reported: Contains-GT 81.67% (file) / 58.33% (elements) / 56.33% (edit locations) pre-repair; the task's headline 69.7/52.0/35.3 is a different, stricter post-repair metric ("% of generated patches whose edited lines are a superset of the ground-truth edit locations") — note this distinction, the two Agentless numbers in circulation are not the same metric. **Transplant verdict:** the skeleton format is the one fully-portable piece (see #1 above); the selection logic is pure LLM judgment with no deterministic analog offered in the paper.

### LocAgent — arXiv:2503.09089
**Narrowing:** deterministic AST-built heterogeneous graph (contains/invoke/import/inherit edges) + three tools (`SearchEntity`, `TraverseGraph`, `RetrieveEntity`) driven by an LLM agent's chain-of-thought. **Deterministic vs LLM:** graph construction and all three tool *mechanics* are deterministic; only *which entity/hop to query next* is LLM judgment — and the ablation shows that judgment matters far less than the paper's framing implies (see #2 above). Reported (SWE-Bench-Lite): File Acc@1 75.91–77.74%, Acc@5 92.70–94.16%; Function Acc@5 71.90–73.36%, Acc@10 77.01–77.37% (Qwen-2.5-32B-ft vs Claude-3.5). **Transplant verdict:** high — see #2, the highest-ROI transplant identified.

### SweRank — arXiv:2505.07849
**Narrowing:** genuinely different architecture — no agent, no graph. A bi-encoder (`SweRankEmbed`, 137M or 7B) embeds the issue text and every function-level code chunk in a shared space and ranks by cosine similarity directly; an optional listwise LLM reranker (`SweRankLLM`) re-orders the top candidates. **Deterministic vs LLM:** `SweRankEmbed` alone has **zero LLM in the loop** — critically, this is the answer to "does anything do well at function-level without an LLM": **SweRankEmbed-Large gets Function Acc@10 = 82.12%** on SWE-Bench-Lite, beating LocAgent-Claude-3.5's 77.37% Acc@10, with no LLM call at inference. It *is* a trained model though (contrastive fine-tuning on 67,341 real GitHub PRs via InfoNCE + hard-negative mining, dataset name SweLoc) — not training-free. `SweRankLLM` (the reranker) reintroduces an LLM and lifts Function Acc@5 to 81.39%/Acc@10 88.69% (Large). **Transplant verdict:** the *architecture lesson* (embed/score at function-chunk granularity, not file or arbitrary-window granularity) directly reinforces #1/#2 above and is the strongest empirical proof that granularity — not reasoning — is most of what closes roust's function-level gap. The weights/training itself is not transplantable to a training-free pipeline, but note this reframes the ceiling question: SweRankEmbed shows a large chunk of what looks like "LLM judgment" in agentic systems is actually learnable lexical/semantic pattern-matching over (issue, buggy-function) pairs — it is inaccessible to roust specifically *because roust forgoes training*, not because it is fundamentally unformalizable.

### CoSIL — arXiv:2503.22424
**Narrowing:** two-phase — module-level call graph (built by parsing import statements, then an LLM textualizes/expands the dependency picture) narrows candidate files, then a function call graph is iteratively BFS-expanded with an LLM-driven relevance pruner (boolean "is this node related to the issue?" call) at each node, up to 10 iterations, plus a final LLM "reflection" pass to clean up formatting. **Deterministic vs LLM:** import parsing is deterministic; the module-graph *construction* itself is LLM-assisted (not pure static analysis, per the paper's own Algorithm 1 — it prompts the LLM to "analyze… and generate" the dependency graph rather than computing it from the AST directly, an odd design choice that forfeits determinism for no clearly stated benefit); the pruner and reflection step are LLM calls. Reported (SWE-bench Lite, Qwen2.5-Coder-32B): Top-1 file 61.3%, function 43.3%; MAP 0.705 (file) / 0.465 (function). Ablation: removing iterative search costs the most (-27.8pp on function Top-1), module-call-graph -11.5pp, reflection -9.9pp, pruning -4.6pp. **Transplant verdict:** the deterministic BFS/expansion scaffold is transplantable (already covered by #3 and roust's roadmap); the graph-construction-via-LLM design and the semantic pruner are not — and notably aren't even validated as the dominant contributor (pruning is the *smallest* ablation effect at -4.6pp), so little is lost by approximating the pruner with a lexical-overlap threshold instead of an LLM call.

### OrcaLoca — arXiv:2502.00350
**Narrowing:** priority-queue agent over a CodeGraph (containment + reference edges): default action priority 1, elevated by LLM-perceived relevance, decremented via a repeat-action counter (deterministic); oversized classes/files get sub-agent LLM scoring of per-method relevance (top-k=3/2); context is pruned to the 12 graph-shortest-path-nearest entries (fully deterministic). **Deterministic vs LLM:** priority *elevation logic* and per-method relevance scoring are LLM judgment; the shortest-path distance metric and the accumulation-counter demotion are deterministic. Reported: File Match 83.33%, Function Match 65.33%, Resolved 41.00% (no Acc@k/line-level breakdown given). Ablation shows the three components (priority scheduling, decomposition, pruning) each contribute a roughly even ~3-5pp when removed — no component dominates, and critically **no pure-LLM-agent baseline is reported**, so the deterministic pruning's true incremental value over "just ask the LLM" is unmeasured in this paper. **Transplant verdict:** the shortest-path CodeGraph distance metric is directly transplantable and independently reinforces #3; the priority-elevation and method-relevance-scoring pieces are LLM judgment with no offered deterministic substitute.

### Newer (2025-2026): SweRank+ and SWE-Adept
**SweRank+** (arXiv:2512.20482) extends SweRank multilingual (10 languages, SweLocMulti dataset) and adds `SweRankAgent` (multi-turn query reformulation on top of the same bi-encoder). Function Acc@10 on SWE-Bench-Lite: SweRankMulti-Large+reranker 89.78%, SweRankAgent 88.32% — both still beat LocAgent-Claude-3.5's 77.37%. Confirms the granularity-over-reasoning pattern rather than adding a new mechanism. **SWE-Adept** (arXiv:2603.01327) is the current apparent leader: deterministic tree-sitter code-structure trees (contains/invokes adjacency, same idea as LocAgent's graph) with **agent-directed depth-first traversal** rather than BFS, running on GPT-5.2/Claude-Sonnet-4.5. Function Acc@5 on SWE-Bench-Lite: 87.8% (SWE-Adept) vs LocAgent 81.8%, OrcaLoca 83.4%, SWE-agent 78.6% — a purely prompting/agent framework, no fine-tuning. It doesn't introduce a new deterministic mechanism beyond what LocAgent/OrcaLoca already established; its gain over them appears to come from frontier-model reasoning quality (Claude-4.5/GPT-5.2 vs Qwen/Claude-3.5), reinforcing that the residual gap at the top of the leaderboard is increasingly judgment-quality-bound, not scaffolding-bound.

**Deterministic, non-LLM, non-trained baseline for calibration:** plain BM25 gets 33.67% file / 13.00% function on SWE-Bench-Lite (BM25-Lucene), and a separate benchmark reports the best deterministic-retrieval configuration reaching only 22.0% Acc@5 at function level — both far below roust's 41.0% FUNCTION, confirming roust's structural/symbol-weighted signal is already meaningfully ahead of naive lexical retrieval; the gap to the LLM/trained systems is specifically in granularity-matched candidate enumeration (#1/#2), not in the base ranking signal being weak.

## Does anything hit function-level well with no LLM?

**Yes: SweRankEmbed** (a trained bi-encoder, arXiv:2505.07849) — Function Acc@10 = 82.12% with zero LLM calls at inference, beating LocAgent-Claude-3.5's agentic 77.37%. It is not training-free (contrastive fine-tuning on 67k real GitHub PRs), so it's not directly available to roust, but it is the strongest evidence in this literature that **granularity of the candidate unit (function-chunk, not file/window) explains more of the file→function accuracy drop than "reasoning" does** — LocAgent's own ablation (SearchEntity >> TraverseGraph) says the same thing from the agentic side.

## Top 3 transplants, ranked by expected function/line gain per unit effort

1. **AST-boundary-snapped candidate/packing units** (Agentless skeleton construction, repurposed) — cheapest (one deterministic parse pass per file, tree-sitter or `ast` module already implied by existing symbol-weighting work), directly targets the metric-definition mismatch between "packed region" and "gold function."
2. **Function/class-granularity BM25 index** (LocAgent's `SearchEntity`, ablation-proven as the dominant lever over graph traversal) — cheap, reuses roust's existing lexical/symbol-weighting machinery, just re-indexed at a finer unit; largest validated effect size (-19pp when removed) of anything reviewed here.
3. **Deterministic call/reference-graph shortest-path re-ranking** (OrcaLoca's distance metric / CoSIL's BFS scaffold stripped of its LLM pruner / roust's own roadmap items #1, #3, #4) — moderate effort (requires a static call/import graph, already planned), secondary because it's a tie-breaker on an already-correct candidate set rather than a fix to the candidate-set granularity itself.

## Where the training-free ceiling likely sits

Two things in this literature look like genuine judgment with no deterministic substitute on offer: (a) **multi-hop causal reasoning connecting a user-visible symptom to a root cause that shares no vocabulary and no direct graph edge with the issue text** — e.g., config-driven or duck-typed indirection, where LocAgent's LLM decides *which* entity to search next based on an inferred causal story, not a lexical or structural cue; and (b) **discriminating among several functions that are all structurally and lexically equidistant from the query but only one of which is semantically wrong** — Agentless's final edit-location stage and OrcaLoca's per-method relevance scoring both delegate exactly this decision to the LLM with no deterministic fallback proposed anywhere in these five papers. However, SweRankEmbed's result tempers how "fundamental" that ceiling really is: a substantial share of what looks like judgment in the agentic systems is recoverable by a *trained* (not reasoned) similarity function over (issue, function) pairs — meaning the honest ceiling for a **training-free** system is probably lower than the honest ceiling for judgment-in-general, and the gap roust can still close with #1-#3 is mostly a granularity/scaffolding gap, not yet a reasoning gap. The reasoning gap becomes the binding constraint only after granularity is fixed — current evidence doesn't let us say how much of roust's 41.0%→52-82% gap is scaffolding-closeable versus judgment-bound until #1/#2 are actually tried.
